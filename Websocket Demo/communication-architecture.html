🖥️ VPS Backend

  • Receives: config updates (drinks, theme, coaster↔animation mapping), animation uploads, orders, interaction logs
  • Outputs: config & animation list to Table & iOS; real-time config push to Table; new order notifications to iOS

🪵 Table (React app)

  • Receives: config & animation metadata (on load); real-time config/theme updates via WebSocket
  • Outputs: orders (when user places drink), interaction logs (what’s on the table, profiles)

📱 iOS App

  • Receives: current config from VPS; new order alerts (real-time); animation list
  • Outputs: config changes (drinks, mapping), new animation uploads (files), theme/vibe changes
HTTP (REST) WebSocket — used for different flows below

🏠 2. Why we use a VPS (not the devices)

  • User config & animations — We need one place to store drink menus, theme/vibe, and coaster↔animation mappings. Staff change these from iOS; the table loads them from the server.
  • Order records per venue — Orders and interaction logs are stored by restaurant/table so we can report and (later) let other devices fetch them. This belongs on the server, not on a single device.
  • Space & management — Config and animations can be large. We don’t want to manage that only on the table or the phone; the VPS acts as the single source of truth and handles storage.

⚙️ 3. HTTP (REST) vs WebSocket — why we use both

🔵 HTTP / REST

  • Simple request → response; easy to debug (browser, Postman).
  • Good for saving & loading data (config, orders, logs, file uploads).
  • No instant push — client must poll or send again to get updates.
  • Not ideal for “change theme and see it on the table in <1s”.

🟢 WebSocket

  • Persistent connection; server can push updates immediately.
  • Perfect for real-time config/theme sync (iOS → table).
  • Not ideal for large file uploads or “must persist to DB” flows (orders).
  • Needs connection handling (reconnect, auth) on server and clients.
✅ Our choice: hybrid

We use REST for: loading/saving config, uploading animations, sending orders and logs. We use WebSocket for: pushing config/theme changes from iOS to the table in real time, and (optional) pushing new orders to the iOS app. So each flow uses the right tool.

📱 iOS ⇄ 🖥️ VPS ⇄ 🪵 Table

WebSocket: real-time config/theme to table; new orders to iOS
